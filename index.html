Featured Deep Dive

Retry Engine

A closer look at the most infrastructure-level thing I built during HNG.

The Problem

Outgoing HTTP requests in most backend systems are synchronous — the caller blocks until the request completes or fails. When the external service is slow, unavailable, or rate-limiting you, that becomes a problem that propagates upward. You need a layer that decouples the caller from the actual network call, handles transient failures automatically, and gives up cleanly when something is truly broken.

Architecture & Request Flow

  1. Client hits POST /request — the request is persisted to the database with status pending and the endpoint returns immediately.
  2. A background worker polls the queue every ~500ms and picks up pending requests.
  3. The worker makes the actual outgoing HTTP call on behalf of the client.
  4. On success, the request is marked completed. On a retryable failure (5xx, timeout, network error), it is re-queued with an incremented attempt count and a calculated next-retry time.
  5. On a terminal failure (any 4xx), the request is immediately marked failed — no further retries, no waste.
  6. If a request exhausts all allowed attempts, it moves to dead-letter storage for inspection.

Key Endpoints

POST /request — enqueue a new outgoing request
GET /request/:id — check status of a specific request
GET /requests — list all requests with their current state
GET /dead-letter — inspect requests that exhausted all retries

One Technical Challenge — Backoff With Jitter

The problem: Pure exponential backoff (1s → 2s → 4s → 8s → 16s) sounds clean, but in a system where multiple failed requests retry at the same time, they all wake up simultaneously and hammer the external service together — exactly when it's already struggling. This is called a thundering herd.

How I solved it: Each retry delay is multiplied by a random jitter factor in the range [0.8, 1.2), re-rolled independently on every attempt. This spreads retries across a window instead of bunching them, so the external service gets a more even load curve during recovery. The jitter is re-rolled per attempt rather than once at enqueue time so the spread stays effective across the full backoff sequence.

Learning Reflection

What HNG actually taught me

Honest version. No filler.

  • I learned the difference between code that works and code that holds up. Before HNG, I was happy if endpoints returned the right response. By Stage 4, I was thinking about query latency, cache hit rates, and what happens when 500,000 rows hit the system at once.
  • Edge cases stopped being afterthoughts. Duplicate rows, partial upload failures, race conditions on concurrent writes, expired tokens — I used to skip these. Now I design for them first.
  • Writing before building changed how I code. The RFC process in Stage 5 felt like overhead until I realised it caught two bad decisions before I wrote a single line. I'll keep doing it.
  • I got more comfortable with systems thinking — not just what a component does, but how it fails, what it does to the components around it when it does, and how to make failures visible through logging and structured error responses.
  • I still have a lot to learn. Distributed systems, proper load testing, deeper Rust — these are on the list. HNG showed me where the ceiling is. Now I know what to climb toward.
